- **Fixed: empty placeholder tiles disappeared entirely in Coming Soon, Why Join, and Our Gallery (2026-07-13, regression from the redesign above, same day).** Each tile's style object set both the `background` shorthand (for the gradient/color fallback) *and* the `backgroundImage` longhand (for an uploaded photo, else `undefined`) side by side. React writes `undefined` style values as an empty string, and setting `style.backgroundImage = ''` *after* `style.background = '<gradient>'` (object key order = the order React applies them) clears the image the shorthand had just set — so with no photo uploaded (the default, untouched state), every tile rendered as a fully invisible box instead of falling back to its gradient/color placeholder. Fixed by dropping the `background` shorthand everywhere and driving both the fallback and the photo through `backgroundImage` alone (`url(...)` or a CSS gradient function both work there); Why Join's solid-color fallback moved to `backgroundColor` instead, which doesn't conflict with `backgroundImage`. Verified via computed styles in-browser (fresh tab, no HMR/cache) that all three sections now show their fallback correctly with no photo set.

- **"Our Gallery" changed from a fixed 8-tile grid/manual-scroll to an unlimited, continuously auto-scrolling marquee (2026-07-16) — supersedes the "8 fixed-id tiles" part of rule 20 above (the CTA banner half of that rule is unchanged).** `content.galleryImages` is no longer a fixed-length array — Admin Console → Content's gallery editor now has an "Add" tile (multi-file picker, appends `{id: 'g-<timestamp>-<random>', image}` entries) alongside each tile's own "Remove" button (splices it out entirely, rather than the old behavior of just clearing that slot's image back to the gradient fallback). On the public page, tiles split into two rows by even/odd index (`GalleryRow` in `PublicHome.jsx`); top row scrolls right-to-left, bottom row left-to-right, via one shared CSS keyframe (`.gallery-track`/`galleryScroll` in `index.css`) played normally vs. `animation-direction: reverse`. Tiles are 240×280 (up from 170×200), and the section now carries the same two blurred radial-gradient glow blobs as the "Coming Soon" section for visual consistency between the two dark strips. Pauses on hover (pointer devices) and freezes entirely under `prefers-reduced-motion: reduce`. If the admin removes every tile, the whole section hides (`content.galleryImages.length > 0` guard) rather than showing an empty dark strip.
- **Fixed: gallery marquee ran out of tiles into visible empty space before looping (2026-07-16, same-day follow-up to the row above).** The first version rendered each row's tile list exactly twice and animated `translateX(-50%)` — seamless only when the doubled list happens to be wider than the viewport. With few tiles (or a wide screen), the doubled track was narrower than the screen, so the marquee visibly ran out of content and snapped back. Fixed by computing `repeatCount` from the live viewport width (`Math.ceil(windowWidth / listWidth) + 2`, minimum 2) so a full extra screen-width of tiles always remains once one list-width has scrolled off, and animating by exactly one list-width in px (a `--gallery-shift` custom property read inside the shared keyframe) instead of a fixed `-50%`, since the loop point is always "one list", not "half of however many times we repeated it." Reactive to window resize via the existing `windowWidth` state (already tracked for the Coming Soon carousel's sizing).
- **Edge fade added to the gallery marquee (2026-07-16, same-day).** Each row's `overflow:hidden` wrapper now carries a `.gallery-viewport` class using the same `mask-image` edge-fade technique as `.coming-soon-viewport`/`.event-rail` (tiles fade to transparent at the left/right edges instead of a hard clip), just at a wider fade distance (72px desktop / 36px under 640px) since the gallery tiles are bigger.

## Design & theme
- **Apple-style motion & material pass across both portals (2026-07-14).** Motion foundations in `index.css` (two shared curves: `--ease-spring` ≈ critically-damped settle for UI, `--ease-sheet` = Apple's sheet curve for the drawer): **(1) Press feedback** — every `button` scales to 0.97 on `:active` with a near-instant press and eased release (feedback on pointer-down, not release). **(2) Tab transitions** — both dashboards wrap their tab content in a `<div key={tab} className="tab-panel">`, so switching tabs remounts the wrapper and plays a 0.3s fade-up (`tabIn`). **(3) Translucent chrome** — the mobile menu trigger bars (`.vendor-tabs-bar`/`.admin-tabs-bar`) are now sticky with `backdrop-filter: blur(18px) saturate(160%)` over a 72% `color-mix` of `--bg-card`, so content scrolls under them. **(4) Card hover lift** — `.admin-cards`/`.event-cards` children lift 2px with a soft shadow on hover (pointer devices only, `@media (hover:hover)`). **(5) Modals** — the shared `Sheet`, `PayModal`, and `Toast` scale-fade in on the spring curve with a separate scrim fade (`scrimIn`). **(6) Drawer gestures** (`MobileNavDrawer.jsx`) — the drawer enters/exits along the same path (slides left, scrim fades with its progress), and supports **swipe-to-close**: 1:1 finger tracking after a 10px hysteresis threshold (taps still work), rubber-band resistance when dragged the wrong way, and a release decision based on **velocity** (fast left flick closes from anywhere; slow release only closes past 40% width) with the exit animated from the current position. **(7) Reduced motion** — `@media (prefers-reduced-motion: reduce)` redefines the movement keyframes as plain cross-fades (and stills the parking-pass pulse); the drawer's JS close path also skips its slide. `prefers-reduced-transparency` makes the sticky bars solid. All animation is compositor-friendly (`transform`/`opacity` only).
- **Site-wide re-theme 2026-07-12**, driven by the same Claude Design handoff: rose (`#A6364E`) + `Playfair Display`/`DM Sans` → warm amber/gold (`#9A5B26`/`#B97434`/`#7A431A`) + `Marcellus`/`Karla`, across every screen (public site, vendor portal, admin dashboard incl. dark/night mode, PDF exports).
- `Marcellus` only ships a single (400) weight — headings that used to be bold Playfair are now `fontWeight: 400` Marcellus, matching the design's own convention.
- CSS custom properties in `src/index.css` (`--bg-*`, `--text-*`, `--border-*`) were remapped to warmer equivalents; the semantic status-tint variables (`--tint-green-*`, `--tint-red-*`, `--tint-amber-*`, `--tint-blue-*` — used for approved/rejected/pending/etc. badges) were left untouched on purpose, since they encode status, not brand color, and happen to not collide with the new brand hex values.
- `App.css` (an unused create-vite scaffolding leftover, never imported) was deleted as part of this pass.
- **Dark-mode input text fix (2026-07-13).** Browsers don't inherit `color` into `<input>`/`<textarea>` by default, so every text field in the admin console rendered with the default black text — invisible against the dark `--bg-card` in night mode, even though the field's border and background did switch correctly. Fixed by adding `color:'var(--text-primary)'` to the shared `inp` style constant (`AdminDashboard.jsx`, used by ~27 fields across Content, Events, and other tabs) plus the handful of one-off styled inputs that had the same gap (Compliance's "new offence type" field, Admin Roles' "create admin" fields). **Deliberately left unfixed:** inputs inside popup/modal dialogs (e.g. the Categories "Add/Edit Category" modal) — out of scope per this pass, modals may need their own separate pass later.

## Portal navigation (2026-07-13 redesign)
- **Sign-in screens (`VendorLogin.jsx`, `AdminLogin.jsx`) no longer render the Sidebar/BottomNav shell.** `App.jsx` now detects `isSignIn` (`vScreen==='login'` or `aScreen==='login'|'reset'`) and routes straight to a new shared `AuthLayout.jsx` — a centered card on a full-bleed radial-gradient background (same phone-frame bypass pattern the public home page already used), instead of the old cramped-in-`.app-shell` layout. Previously the sign-in forms sat inside the same fixed-width phone frame as every other screen, with the Sidebar visible (and empty of anything useful) behind them.
- **"Back to home" button** is built into `AuthLayout.jsx` itself (top-left, fixed position) so both vendor and admin sign-in carry it automatically — this is also the fix for "signed-out users need a way back to the public site": Sign out (`VendorDashboard.jsx`/`AdminDashboard.jsx`) already dropped users on the login screen, which previously had no home link at all.
- **`AuthLayout`'s `dark` prop follows the real admin `darkMode` toggle** (`view==='admin' && darkMode`), not a hardcoded true — so admin sign-in respects day/night mode the same way the rest of the console does, and text colors (which use `var(--text-primary)` etc.) stay correctly paired with the card background instead of risking the same invisible-text bug fixed above.
- **Vendor "Forgot password?" flow added (`VendorLogin.jsx`)** — vendor login previously had no reset path at all (unlike admin, which already had a `mustReset` first-sign-in flow). It's a 3-state mock (`login` → `forgot` → `sent`): clicking "Forgot password?" next to the password field swaps to an email-entry screen, submitting shows a "Check your email" confirmation. **No real email is sent** — this matches the app's existing "Phase 1, mock data only" pattern (see admin's own "Forgot passwords are reset to the default by a super admin" messaging), it's a placeholder for a real reset flow once Supabase auth lands.
- **Sidebar's "NAVIGATE" section (Home / Apply as Vendor / Vendor Portal / Admin Console) is now hidden once inside a portal** (`Sidebar.jsx`, `!isVendor && !isAdmin` guard) — previously it showed above the contextual "MY PORTAL"/"CONSOLE" tab list even while already inside that exact portal, duplicating the "Vendor Portal"/"Admin Console" entry as a permanently-active, functionally dead link. It still shows on the vendor registration screen, where there's no portal context yet and the 4-item nav is the only way to move between Home/Apply/Portal/Admin.
- **Mobile tab navigation replaced: wrapping pill row → slide-in drawer.** `VendorDashboard.jsx` (10 tabs) and `AdminDashboard.jsx` (up to 18 tabs, role-filtered) used to render every tab as a `flex-wrap` pill inside `.vendor-tabs-bar`/`.admin-tabs-bar` — fine at 3-4 tabs, but wrapped into 3-5 cramped rows once the full list showed. Both now render a single full-width "current tab" trigger button (hamburger icon + active tab label + chevron) in that same CSS-class slot (still `display:none` ≥768px, since desktop uses the Sidebar), which opens a new shared `MobileNavDrawer.jsx` — a left slide-in panel (`drawerIn` keyframe, `src/index.css`) listing every tab, closes on selection or backdrop tap. Admin's drawer is always dark-themed (`dark` prop hardcoded true), matching the Sidebar's own admin styling being fixed dark regardless of the day/night toggle.
- **New icon:** `menu` (hamburger) added to `Icon.jsx` for the mobile drawer trigger.
- **Vendor registration also moved into `AuthLayout` (same session, follow-up).** The 4-step register form was the last screen still rendering inside the phone-frame shell with the Sidebar (and its NAVIGATE section) beside it. Now `App.jsx` routes `vScreen==='register'` through the same sidebar-less `AuthLayout` as the sign-in screens, with a wider card (`maxWidth` prop, 780 vs the sign-in screens' 420) to fit the form's 2-column desktop grid. The card now uses auto vertical margins instead of flex centering — centered when it fits the viewport, top-anchored and scrollable when taller (plain `align-items:center` would clip the top of the long form). With this, the Sidebar's NAVIGATE section has no remaining screen to show on — `Sidebar` now only ever renders the contextual MY PORTAL/CONSOLE tab lists inside the two dashboards.
- **Portal tab order is a super-admin global setting (2026-07-13, same session — reworked from an earlier per-user version the same day).** Initially each user could drag-reorder their own sidebar/drawer tabs; the user clarified the intent: **one arrangement, set centrally, applies to everyone.** Now the only reorder UI lives in **Admin Console → Settings → "Portal tab order"** (rendered for super admins only — staff admins see just the three toggles), with two cards: "Admin console tabs" (full `ADMIN_TABS` list incl. the super-only Admin Roles — a staff admin's RBAC-hidden tabs keep their assigned position) and "Vendor portal tabs" (`VENDOR_TABS`). Each card's rows can be dragged (dragged row dims, drop target shows an amber insert line) **or** nudged with per-row up/down arrows (touch-friendly), plus a "Reset to default" link that reverts to the code-defined order. The arranged order (`vTabOrder`/`aTabOrder` in the store, arrays of tab ids, `null` = default) is what every admin's sidebar/drawer and every vendor's portal render via `orderTabs()` (`helpers.js`; ids missing from a saved order — e.g. tabs added later in code — keep their code position at the end); vendors and staff admins have **no reorder controls anywhere**. `vTabOrder`/`aTabOrder` are also in `EDIT_SET_KEYS` so a view-only staff admin can't change them programmatically. **Mock-data caveat:** the order persists to localStorage (`sulap_vtab_order`/`sulap_atab_order`, same mechanism as dark mode) — a real backend is what would make it genuinely global across devices; on this mock build it's global across all roles *on the same browser*. Also from this work: `VENDOR_TABS` was deduplicated to live only in `VendorDashboard.jsx` (exported, same pattern as `ADMIN_TABS`); `Sidebar.jsx` imports it.
- **`BottomNav.jsx` deleted (same session, follow-up).** The mobile Home/Apply/Portal/Admin bottom bar became fully redundant: inside a portal the drawer + Sign out cover navigation, sign-in screens use `AuthLayout`, the public home renders standalone, and the register screen already hid it before this redesign — so no screen rendered it anymore. Removed the component, its `App.jsx` usage (`showNav`), and the `.bottom-nav` desktop-hide rule in `index.css`.

## Vendor side

### Vendor Registration (`VendorRegister.jsx`)
4-step form: business details + category + password → contact/social/logistics → product photos → review & accept market terms.
- **Fully wired (as of 2026-07-11):** all fields are controlled, bound to `state.rf`. Step 1 validates business name/owner/email/phone, category selection, and an 8+ char password before advancing. Photo upload is a mock tile counter (up to 8), not real file upload.

### Vendor Login (`VendorLogin.jsx`)
Mock login — no real auth yet, always resolves to a hardcoded "current vendor" (`CURRENT_VENDOR_ID = 'v1'` in `mockData.js`) rather than checking a real password. **Status-gated (as of 2026-07-12):** login now checks that vendor's registration `status` before entering the dashboard — `pending` shows "still under review", `rejected` and `suspended` show their own blocking messages naming the reason, and only `status === 'approved'` reaches the dashboard. ### Booth sharing rule (apply flow, `Modals.jsx` → `ApplyModal`)
When a vendor applies to an event, they choose solo or shared booth:
- A shared booth can have **up to 3 vendors total** (the applicant + up to 2 partners)
- Booth partners must be the **same tier** — an F&B vendor can only share with other F&B vendors, and a non-F&B vendor only with other non-F&B vendors
- **Booth partners must be `approved` vendors** (fixed 2026-07-11 — the partner search previously let `pending`/`suspended` vendors show up and be picked; now filtered out alongside the tier check)
- Partners are picked by searching existing registered vendors (live search-as-you-type on business name, up to 5 results)

## Key business rules

1. **Registration ≠ event approval.** A vendor's `status` (`pending` → `approved`/`rejected`, or `approved` → `suspended`) only reflects whether they're allowed to *be* a Sulap vendor at all. It says nothing about payment or event participation. An approved vendor's individual event applications still each need their own admin decision — event application status (`pending`/`approved`/`rejected` on the application record) is a **separate status from vendor registration status**, even though both reuse the same three words. To keep them from being confused for each other (fixed 2026-07-11): the admin Event Applications list drops the badge entirely and uses buttons instead (see that tab's row above); the vendor's own My Applications tab labels its badge "Awaiting review" instead of "Pending" for the equivalent state.
2. **Vendor must be `approved` before they can apply to any event, or appear in booth-sharing search.** A vendor still sitting in the Vendor Applications queue (`pending`) has zero event-side visibility — no Apply access of their own, and other vendors can't find or add them as a booth partner. Enforced in `VendorDashboard.jsx` (apply gate) and the partner search in `Modals.jsx` → `ApplyModal` (`v.status === 'approved'` filter, fixed 2026-07-11 — see mock data note below).
3. **Deposit and payment status belong to event participation**, not registration — they must never appear on the registration review screen (Vendor Applications / Vendor Listing "view details" modal).
4. **Deposit is per-vendor, not per-event**, and auto-attaches to the vendor's first unpaid event invoice.
5. **Booth sharing** is capped at 3 vendors and restricted to same-tier (F&B/non-F&B) partners.
6. **Parking serial entry is time-locked** to the event's actual dates unless manually overridden by admin.
7. **Suspend and Reject are both reversible.** Suspend only appears on already-approved vendors, is a small deliberately low-emphasis link (not a full button) behind a confirm dialog, and sets `status: 'suspended'` — vendor loses event-application access but stays visible in Vendor Listing with a "Reinstate vendor" option. Rejected vendors are hidden by default under a collapsed "Show N rejected applications" section at the bottom of Vendor Applications, with a "Reconsider" action that moves them back to `'pending'`.
8. **All popups/modals across both admin and vendor are centered dialogs**, not bottom sheets — as of 2026-07-13, `Sheet` (`components/Modals.jsx`, the shared wrapper behind every modal except the admin Category editor and `App.jsx`'s `PayModal`, both already centered) always centers; it previously defaulted to a bottom sheet unless a `centered` prop was passed, which every vendor-facing modal (Apply, Vendor Pass, Deposit, Refund, Doc Preview) omitted. Originally this rule only covered the admin-only vendor/event-application/event-edit modals — now it applies uniformly everywhere.
9. **Events close to applications automatically** once today's date passes the event's `lastApp` date — enforced both on the public home page and inside the vendor dashboard's apply button.
10. **Admin card lists are 2 columns on desktop, 1 on mobile** (`.admin-cards` CSS class, `AdminDashboard.jsx`, as of 2026-07-11) — originally applied to every top-level paginated/browsable card list in the admin console. Deliberately *not* applied to the Activity feed (a chronological timeline with connecting lines — 2 columns would break reading order) or the Vendor Chart ranking (a single card with an internal ranked list, not a list of cards). As of rule 37 (2026-07-16), `.admin-cards` is only still used by **Events** (its existing-events rail predates and isn't part of the table-style rollout) and **Settings**' "Portal tab order" cards — every vendor-listing tab (Vendor Applications, Vendor Listing, Profile Requests, Event Applications, Payments, Deposit Record, Parking, Event Pictures, Vendor Pass, Compliance's Vendor review sub-tab, Categories) has moved onto the shared `TableShell` glass header-row + grid-row table instead. The class itself is untouched in `index.css` in case a future tab needs it again.
11. **Second payment advice + overpayment/refund flow (as of 2026-07-11).** When admin marks a vendor's event payment `partial`, the vendor's Payments tab gains a second "Upload second payment advice (remaining balance)" button (`payments[key].advice2`) — admin can then download it too, mirroring the first. Separately, if a vendor is overpaid (`paid > total due`, computed live, not stored), both sides show an "Overpaid by RM X" banner. Admin gets an **Arrange refund** action (`RefundModal` in `Modals.jsx`) requiring reference code + date + time before it can be marked complete (`refunds[vendorId-eventId]`, `MERGE_REFUNDS`); once complete, admin gets a **Close case** button. The vendor sees the same refund status read-only (in progress → completed → closed) so both sides always agree on where the refund stands — closing doesn't delete the record, it just stops showing the action buttons and switches to a muted "Refund closed" line on both sides.
12. **Payment status is derived from an amount, never asserted directly (as of 2026-07-11).** The old "Mark: Paid / Partial / Unpaid" buttons let admin flip the status label without ever recording how much was actually received (the standalone "Paid" button didn't even open the amount modal). Replaced with a single **Record payment** button (pre-filled with the outstanding total for a one-click exact match) that always takes an amount and derives status: amount = total → `paid`; amount < total → `partial`; amount > total → `paid` + triggers the overpaid/refund flow (rule 11); amount = 0 → `unpaid`. A separate **Reset to unpaid** button handles corrections without opening the modal. This is the actual live `PayModal`, defined in `App.jsx` — a near-identical but unused duplicate `PayModal` in `Modals.jsx` was deleted in the same pass (dead code that had silently drifted out of sync with the one actually rendered).
13. **Compliance hold blocks Approve on the Shortlist tab.** A shortlisted vendor with unresolved offences in the last `skipN` markets (settings-configurable "skip after N offences" window) cannot be approved directly — `complianceHold(vendorId, eventId)` in `AdminDashboard.jsx` returns their matching offences, and if any exist and no override is on file (`compOverrides[vendorId-eventId]`), the row shows an amber hold banner and swaps the Approve button for **Override hold**, which sets `compOverrides[...] = true` (logged + toasted) and unlocks the normal Approve button — it does not clear or resolve the underlying offences, only permits this one approval.
14. **Release vendor (added 2026-07-12) reverses an approval, same target state as Reject.** Once an application reaches `status === 'approved'` on the Shortlist tab, no other action existed to undo it (a prior session's fix, commit `f71666b`, intentionally removed the old "Remove" button so approved vendors "stay" in the Shortlist as the record for that market). For last-minute vendor cancellations, a **Release vendor** text link is available — deliberately low-emphasis (underlined text, no background, red tint) rather than a full button, matching the existing Suspend-vendor pattern (rule 7), since this is a rare, consequential action that shouldn't visually compete with the primary Approve/Reject flow. **Positioned in the button row next to "View & share booth" (fixed 2026-07-12), not beside the name/Approved label** — an earlier placement wrapped inconsistently at narrow widths depending on vendor name length and whether the "Sharing" tag was present. Behind a `window.confirm` guard naming the vendor; on confirm it sets `status: 'pending'` — identical destination to the shortlist Reject action — so the vendor reappears in the Applications tab exactly as if never decided, rather than being deleted or left in limbo. Logs via `logActivity` and shows a toast, matching every other status transition in this tab.
15. **Application detail modal (`AppDetailModal` in `Modals.jsx`) now links to the applying vendor's own full profile, not just partners' (added 2026-07-13).** Previously the booth-sharing section only made *partner* rows clickable through to their full `VendorDetailModal` profile (contact info, e-invoice/bank details, product photos, Suspend action, etc.) — the main applicant (`v`) had no equivalent link, so a solo-booth application showed only the inline summary (name, description, photos) with no way to reach the vendor's full record. Fixed by adding a clickable row directly under the status badge — same visual pattern as a partner row (icon + name + chevron on a tinted background) — that opens `vendorDetailId: v.id` with `vendorDetailReturnAppId: appDetailId` set, so closing the vendor profile returns to this application detail rather than closing everything. This applies uniformly whether the application is solo or shared; a shared application now shows both the main vendor's link (new) and the partner's link (pre-existing) side by side.

15. **Category data model (as of 2026-07-12): `{id, name, description, icon}`.** Previously `INITIAL_CATS` had only `{id, name}` (bare structure) while `VendorRegister.jsx` maintained its own hardcoded duplicate with full metadata. Now unified: a single `INITIAL_CATS` in mockData provides all fields. VendorRegister pulls `cats` from store instead of hardcoded; admin's Categories tab can add categories with the modal providing name + description + icon picker. All categories now sync across vendor sign-up and admin console — add a category in admin, it appears in the sign-up form immediately.
16. **E-Invoice & bank details are mandatory before a vendor can apply to any market (added 2026-07-12).** Once a vendor is `approved`, their Profile tab (`VendorDashboard.jsx`) gains an "E-Invoice & bank details" card — 8 required fields: Company name (per SSM), Company Registration Number (new, 12 digits), Tax Identification Number (TIN, no spacing), SST Number (or "N/A" if not SST-registered), Business registration address, and Bank Name / Account Number / Account Holder Name (for deposit refunds). `einvoiceComplete(vendor)` (`lib/helpers.js`) checks all 8 fields are non-empty on `vendor.einvoice`. The Available Markets apply button checks `vendorApproved` **and then** `einvoiceOk` — incomplete vendors see "Complete E-Invoice info to apply" instead of the normal Apply/Applications-closed states. Data lives on the vendor record as `vendor.einvoice = {companyName, regNo, tin, sstNo, regAddress, bankName, bankAccNo, bankHolder}`, seeded empty (`EMPTY_EINVOICE` in `mockData.js`) for new registrations. **`v1` (Nutmeg & Clay, the mock "current vendor") is seeded pre-filled** — it originally started empty to demo the gate live, but that was inconsistent with its own seed data already having an *approved* application to Tamu Weekend Bazaar (`INITIAL_APPS` entry `a2`), which shouldn't have been possible under this rule. The gate itself only applies going forward (it doesn't retroactively revoke pre-existing approvals), but the seed data now reads as compliant rather than contradictory. Borneo Brews and Pulau Soap Co. also carry pre-filled sample data; `v3`/`v5`/`v6`/`v7` remain empty to demonstrate the "Incomplete" state in the admin Vendor Listing modal. **As of the request-flow change (rule 17), the vendor no longer writes `vendor.einvoice` directly** — their submission goes through the same admin-approval queue as other locked fields, and `einvoiceOk`/the apply gate only reflect data admin has approved.
17. **Vendor Profile fields split into directly-editable vs. admin-approval-required (added 2026-07-12).** Only **Instagram, Facebook, TikTok, and Power supply needs** are directly vendor-editable — saved instantly via `MERGE_VENDORS` from the Profile tab's "Social media & power supply" card, no review needed. **Everything else vendor might want to change goes through a request queue**: the "Vendor details" card (Brand name, Contact person, Category, Email, Phone, Car plate number, Product description) and the "E-Invoice & bank details" card (all 8 fields, including a vendor's *first-ever* submission — see rule 16) both use a "Request a change"/"Complete now" flow. Submitting pushes a record onto `state.profileRequests` (`{id, vendorId, section: 'details'|'einvoice', changes, submittedAt, status:'pending'}`, `MERGE_PROFILE_REQUESTS` in `lib/store.jsx`) instead of writing the vendor record directly; the Profile tab shows an amber "pending admin review" banner and blocks a second request for that section until the first is decided. **Admin has two ways to act on this:** (1) a dedicated **Profile Requests** tab (queue view, approve/reject with an old→new diff) or (2) inline at the top of the Vendor Listing detail modal when opened for that vendor. 19. **Popup/scrollbar redesign (2026-07-13):** every scrollable popup (all `Sheet`-based modals in `Modals.jsx`, the admin Category editor, and the admin sidebar's tab-nav list) now uses a thin, theme-colored scrollbar (`.themed-scroll` / `.themed-scroll-light` in `index.css`, using `scrollbar-color`/`::-webkit-scrollbar-thumb`) instead of the browser's default gray one. Replaced with `components/RichTextEditor.jsx`, a themed wrapper around `react-quill-new` (Quill, React-19-compatible fork) with a toolbar limited to headings, bold/italic/underline, and ordered/bullet lists — enough for long formatted policy text without a full document-editor's surface area. `content.terms` now stores Quill's HTML output directly; `INITIAL_CONTENT.terms` in `mockData.js` was converted from the old plain-text template to equivalent HTML (`<p>`/`<strong>` per section).
