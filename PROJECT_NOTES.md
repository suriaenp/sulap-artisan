- **Fixed: empty placeholder tiles disappeared entirely in Coming Soon, Why Join, and Our Gallery (2026-07-13, regression from the redesign above, same day).** Each tile's style object set both the `background` shorthand (for the gradient/color fallback) *and* the `backgroundImage` longhand (for an uploaded photo, else `undefined`) side by side. React writes `undefined` style values as an empty string, and setting `style.backgroundImage = ''` *after* `style.background = '<gradient>'` (object key order = the order React applies them) clears the image the shorthand had just set ŌĆö so with no photo uploaded (the default, untouched state), every tile rendered as a fully invisible box instead of falling back to its gradient/color placeholder. Fixed by dropping the `background` shorthand everywhere and driving both the fallback and the photo through `backgroundImage` alone (`url(...)` or a CSS gradient function both work there); Why Join's solid-color fallback moved to `backgroundColor` instead, which doesn't conflict with `backgroundImage`. Verified via computed styles in-browser (fresh tab, no HMR/cache) that all three sections now show their fallback correctly with no photo set.

- **Fixed: gallery marquee ran out of tiles into visible empty space before looping (2026-07-16, same-day follow-up to the row above).** The first version rendered each row's tile list exactly twice and animated `translateX(-50%)` ŌĆö seamless only when the doubled list happens to be wider than the viewport. With few tiles (or a wide screen), the doubled track was narrower than the screen, so the marquee visibly ran out of content and snapped back. Fixed by computing `repeatCount` from the live viewport width (`Math.ceil(windowWidth / listWidth) + 2`, minimum 2) so a full extra screen-width of tiles always remains once one list-width has scrolled off, and animating by exactly one list-width in px (a `--gallery-shift` custom property read inside the shared keyframe) instead of a fixed `-50%`, since the loop point is always "one list", not "half of however many times we repeated it." Reactive to window resize via the existing `windowWidth` state (already tracked for the Coming Soon carousel's sizing).
- **Edge fade added to the gallery marquee (2026-07-16, same-day).** Each row's `overflow:hidden` wrapper now carries a `.gallery-viewport` class using the same `mask-image` edge-fade technique as `.coming-soon-viewport`/`.event-rail` (tiles fade to transparent at the left/right edges instead of a hard clip), just at a wider fade distance (72px desktop / 36px under 640px) since the gallery tiles are bigger.

## Design & theme
- **Apple-style motion & material pass across both portals (2026-07-14).** Motion foundations in `index.css` (two shared curves: `--ease-spring` Ōēł critically-damped settle for UI, `--ease-sheet` = Apple's sheet curve for the drawer): **(1) Press feedback** ŌĆö every `button` scales to 0.97 on `:active` with a near-instant press and eased release (feedback on pointer-down, not release). **(2) Tab transitions** ŌĆö both dashboards wrap their tab content in a `<div key={tab} className="tab-panel">`, so switching tabs remounts the wrapper and plays a 0.3s fade-up (`tabIn`). **(3) Translucent chrome** ŌĆö the mobile menu trigger bars (`.vendor-tabs-bar`/`.admin-tabs-bar`) are now sticky with `backdrop-filter: blur(18px) saturate(160%)` over a 72% `color-mix` of `--bg-card`, so content scrolls under them. **(4) Card hover lift** ŌĆö `.admin-cards`/`.event-cards` children lift 2px with a soft shadow on hover (pointer devices only, `@media (hover:hover)`). **(5) Modals** ŌĆö the shared `Sheet`, `PayModal`, and `Toast` scale-fade in on the spring curve with a separate scrim fade (`scrimIn`). **(6) Drawer gestures** (`MobileNavDrawer.jsx`) ŌĆö the drawer enters/exits along the same path (slides left, scrim fades with its progress), and supports **swipe-to-close**: 1:1 finger tracking after a 10px hysteresis threshold (taps still work), rubber-band resistance when dragged the wrong way, and a release decision based on **velocity** (fast left flick closes from anywhere; slow release only closes past 40% width) with the exit animated from the current position. **(7) Reduced motion** ŌĆö `@media (prefers-reduced-motion: reduce)` redefines the movement keyframes as plain cross-fades (and stills the parking-pass pulse); the drawer's JS close path also skips its slide. `prefers-reduced-transparency` makes the sticky bars solid. All animation is compositor-friendly (`transform`/`opacity` only).
- **Site-wide re-theme 2026-07-12**, driven by the same Claude Design handoff: rose (`#A6364E`) + `Playfair Display`/`DM Sans` ŌåÆ warm amber/gold (`#9A5B26`/`#B97434`/`#7A431A`) + `Marcellus`/`Karla`, across every screen (public site, vendor portal, admin dashboard incl. dark/night mode, PDF exports).
- `Marcellus` only ships a single (400) weight ŌĆö headings that used to be bold Playfair are now `fontWeight: 400` Marcellus, matching the design's own convention.
- CSS custom properties in `src/index.css` (`--bg-*`, `--text-*`, `--border-*`) were remapped to warmer equivalents; the semantic status-tint variables (`--tint-green-*`, `--tint-red-*`, `--tint-amber-*`, `--tint-blue-*` ŌĆö used for approved/rejected/pending/etc. badges) were left untouched on purpose, since they encode status, not brand color, and happen to not collide with the new brand hex values.
- `App.css` (an unused create-vite scaffolding leftover, never imported) was deleted as part of this pass.
- **Dark-mode input text fix (2026-07-13).** Browsers don't inherit `color` into `<input>`/`<textarea>` by default, so every text field in the admin console rendered with the default black text ŌĆö invisible against the dark `--bg-card` in night mode, even though the field's border and background did switch correctly. Fixed by adding `color:'var(--text-primary)'` to the shared `inp` style constant (`AdminDashboard.jsx`, used by ~27 fields across Content, Events, and other tabs) plus the handful of one-off styled inputs that had the same gap (Compliance's "new offence type" field, Admin Roles' "create admin" fields). **Deliberately left unfixed:** inputs inside popup/modal dialogs (e.g. the Categories "Add/Edit Category" modal) ŌĆö out of scope per this pass, modals may need their own separate pass later.

## Key business rules

1. **Registration ŌēĀ event approval.** A vendor's `status` (`pending` ŌåÆ `approved`/`rejected`, or `approved` ŌåÆ `suspended`) only reflects whether they're allowed to *be* a Sulap vendor at all. It says nothing about payment or event participation. An approved vendor's individual event applications still each need their own admin decision ŌĆö event application status (`pending`/`approved`/`rejected` on the application record) is a **separate status from vendor registration status**, even though both reuse the same three words. To keep them from being confused for each other (fixed 2026-07-11): the admin Event Applications list drops the badge entirely and uses buttons instead (see that tab's row above); the vendor's own My Applications tab labels its badge "Awaiting review" instead of "Pending" for the equivalent state.
2. **Vendor must be `approved` before they can apply to any event, or appear in booth-sharing search.** A vendor still sitting in the Vendor Applications queue (`pending`) has zero event-side visibility ŌĆö no Apply access of their own, and other vendors can't find or add them as a booth partner. Enforced in `VendorDashboard.jsx` (apply gate) and the partner search in `Modals.jsx` ŌåÆ `ApplyModal` (`v.status === 'approved'` filter, fixed 2026-07-11 ŌĆö see mock data note below).
3. **Deposit and payment status belong to event participation**, not registration ŌĆö they must never appear on the registration review screen (Vendor Applications / Vendor Listing "view details" modal).
4. **Deposit is per-vendor, not per-event**, and auto-attaches to the vendor's first unpaid event invoice.
5. **Booth sharing** is capped at 3 vendors and restricted to same-tier (F&B/non-F&B) partners.
6. **Parking serial entry is time-locked** to the event's actual dates unless manually overridden by admin.
7. **Suspend and Reject are both reversible.** Suspend only appears on already-approved vendors, is a small deliberately low-emphasis link (not a full button) behind a confirm dialog, and sets `status: 'suspended'` ŌĆö vendor loses event-application access but stays visible in Vendor Listing with a "Reinstate vendor" option. Rejected vendors are hidden by default under a collapsed "Show N rejected applications" section at the bottom of Vendor Applications, with a "Reconsider" action that moves them back to `'pending'`.
8. **All popups/modals across both admin and vendor are centered dialogs**, not bottom sheets ŌĆö as of 2026-07-13, `Sheet` (`components/Modals.jsx`, the shared wrapper behind every modal except the admin Category editor and `App.jsx`'s `PayModal`, both already centered) always centers; it previously defaulted to a bottom sheet unless a `centered` prop was passed, which every vendor-facing modal (Apply, Vendor Pass, Deposit, Refund, Doc Preview) omitted. Originally this rule only covered the admin-only vendor/event-application/event-edit modals ŌĆö now it applies uniformly everywhere.
9. **Events close to applications automatically** once today's date passes the event's `lastApp` date ŌĆö enforced both on the public home page and inside the vendor dashboard's apply button.
10. **Admin card lists are 2 columns on desktop, 1 on mobile** (`.admin-cards` CSS class, `AdminDashboard.jsx`, as of 2026-07-11) ŌĆö originally applied to every top-level paginated/browsable card list in the admin console. Deliberately *not* applied to the Activity feed (a chronological timeline with connecting lines ŌĆö 2 columns would break reading order) or the Vendor Chart ranking (a single card with an internal ranked list, not a list of cards). As of rule 37 (2026-07-16), `.admin-cards` is only still used by **Events** (its existing-events rail predates and isn't part of the table-style rollout) and **Settings**' "Portal tab order" cards ŌĆö every vendor-listing tab (Vendor Applications, Vendor Listing, Profile Requests, Event Applications, Payments, Deposit Record, Parking, Event Pictures, Vendor Pass, Compliance's Vendor review sub-tab, Categories) has moved onto the shared `TableShell` glass header-row + grid-row table instead. The class itself is untouched in `index.css` in case a future tab needs it again.
11. **Second payment advice + overpayment/refund flow (as of 2026-07-11).** When admin marks a vendor's event payment `partial`, the vendor's Payments tab gains a second "Upload second payment advice (remaining balance)" button (`payments[key].advice2`) ŌĆö admin can then download it too, mirroring the first. Separately, if a vendor is overpaid (`paid > total due`, computed live, not stored), both sides show an "Overpaid by RM X" banner. Admin gets an **Arrange refund** action (`RefundModal` in `Modals.jsx`) requiring reference code + date + time before it can be marked complete (`refunds[vendorId-eventId]`, `MERGE_REFUNDS`); once complete, admin gets a **Close case** button. The vendor sees the same refund status read-only (in progress ŌåÆ completed ŌåÆ closed) so both sides always agree on where the refund stands ŌĆö closing doesn't delete the record, it just stops showing the action buttons and switches to a muted "Refund closed" line on both sides.
12. **Payment status is derived from an amount, never asserted directly (as of 2026-07-11).** The old "Mark: Paid / Partial / Unpaid" buttons let admin flip the status label without ever recording how much was actually received (the standalone "Paid" button didn't even open the amount modal). Replaced with a single **Record payment** button (pre-filled with the outstanding total for a one-click exact match) that always takes an amount and derives status: amount = total ŌåÆ `paid`; amount < total ŌåÆ `partial`; amount > total ŌåÆ `paid` + triggers the overpaid/refund flow (rule 11); amount = 0 ŌåÆ `unpaid`. A separate **Reset to unpaid** button handles corrections without opening the modal. This is the actual live `PayModal`, defined in `App.jsx` ŌĆö a near-identical but unused duplicate `PayModal` in `Modals.jsx` was deleted in the same pass (dead code that had silently drifted out of sync with the one actually rendered).
13. **Compliance hold blocks Approve on the Shortlist tab.** A shortlisted vendor with unresolved offences in the last `skipN` markets (settings-configurable "skip after N offences" window) cannot be approved directly ŌĆö `complianceHold(vendorId, eventId)` in `AdminDashboard.jsx` returns their matching offences, and if any exist and no override is on file (`compOverrides[vendorId-eventId]`), the row shows an amber hold banner and swaps the Approve button for **Override hold**, which sets `compOverrides[...] = true` (logged + toasted) and unlocks the normal Approve button ŌĆö it does not clear or resolve the underlying offences, only permits this one approval.
14. **Release vendor (added 2026-07-12) reverses an approval, same target state as Reject.** Once an application reaches `status === 'approved'` on the Shortlist tab, no other action existed to undo it (a prior session's fix, commit `f71666b`, intentionally removed the old "Remove" button so approved vendors "stay" in the Shortlist as the record for that market). For last-minute vendor cancellations, a **Release vendor** text link is available ŌĆö deliberately low-emphasis (underlined text, no background, red tint) rather than a full button, matching the existing Suspend-vendor pattern (rule 7), since this is a rare, consequential action that shouldn't visually compete with the primary Approve/Reject flow. **Positioned in the button row next to "View & share booth" (fixed 2026-07-12), not beside the name/Approved label** ŌĆö an earlier placement wrapped inconsistently at narrow widths depending on vendor name length and whether the "Sharing" tag was present. Behind a `window.confirm` guard naming the vendor; on confirm it sets `status: 'pending'` ŌĆö identical destination to the shortlist Reject action ŌĆö so the vendor reappears in the Applications tab exactly as if never decided, rather than being deleted or left in limbo. Logs via `logActivity` and shows a toast, matching every other status transition in this tab.
15. **Application detail modal (`AppDetailModal` in `Modals.jsx`) now links to the applying vendor's own full profile, not just partners' (added 2026-07-13).** Previously the booth-sharing section only made *partner* rows clickable through to their full `VendorDetailModal` profile (contact info, e-invoice/bank details, product photos, Suspend action, etc.) ŌĆö the main applicant (`v`) had no equivalent link, so a solo-booth application showed only the inline summary (name, description, photos) with no way to reach the vendor's full record. Fixed by adding a clickable row directly under the status badge ŌĆö same visual pattern as a partner row (icon + name + chevron on a tinted background) ŌĆö that opens `vendorDetailId: v.id` with `vendorDetailReturnAppId: appDetailId` set, so closing the vendor profile returns to this application detail rather than closing everything. This applies uniformly whether the application is solo or shared; a shared application now shows both the main vendor's link (new) and the partner's link (pre-existing) side by side.

15. **Category data model (as of 2026-07-12): `{id, name, description, icon}`.** Previously `INITIAL_CATS` had only `{id, name}` (bare structure) while `VendorRegister.jsx` maintained its own hardcoded duplicate with full metadata. Now unified: a single `INITIAL_CATS` in mockData provides all fields. VendorRegister pulls `cats` from store instead of hardcoded; admin's Categories tab can add categories with the modal providing name + description + icon picker. All categories now sync across vendor sign-up and admin console ŌĆö add a category in admin, it appears in the sign-up form immediately.
16. **E-Invoice & bank details are mandatory before a vendor can apply to any market (added 2026-07-12).** Once a vendor is `approved`, their Profile tab (`VendorDashboard.jsx`) gains an "E-Invoice & bank details" card ŌĆö 8 required fields: Company name (per SSM), Company Registration Number (new, 12 digits), Tax Identification Number (TIN, no spacing), SST Number (or "N/A" if not SST-registered), Business registration address, and Bank Name / Account Number / Account Holder Name (for deposit refunds). `einvoiceComplete(vendor)` (`lib/helpers.js`) checks all 8 fields are non-empty on `vendor.einvoice`. The Available Markets apply button checks `vendorApproved` **and then** `einvoiceOk` ŌĆö incomplete vendors see "Complete E-Invoice info to apply" instead of the normal Apply/Applications-closed states.
